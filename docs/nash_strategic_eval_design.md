## 1. 概要

現在の Nash CPU は「この手を打つと盤面がどれだけ良くなるか」を一手シミュレーションして最善手を選ぶ。
そのため**その場の最善手**は打てるが、**キャラごとの勝ち筋（勝利条件）**に沿った立ち回りができていない。

本設計は、CPU の盤面採点（evalState）に**二軸の戦略項**を足すことで、
「自分の勝ち筋を進める手／相手の勝ち筋を止める手」を自然に高評価させる。
キャラ別の手打ち（ハードコードしたマッチアップ分岐）は増やさず、**原理から正しい立ち回りを創発**させることを狙う。

- **軸A**：自分の勝ち筋を促進する（自分の強みを活かす）
- **軸B**：相手の勝ち筋を抑制する（相手の強みを消す）
- 人間は局面に応じて A・B・両方を使い分ける。CPU にも同じ判断をさせる。

---

## 2. 狙い／プレイヤー体験

- **強いけれど、毎回同じではない。** 読み合いで勝てる余地を必ず残す。ただし勝てる時は確実に勝つ。
- 各キャラが「らしい勝ち方」をする＝対戦相手として理解可能で、攻略しがいがある。
- 「正解暗記マシン」化を避ける（同じ手の連打でバレる状態を作らない）。

## 3. 前提：正本照合で確定した6キャラの勝ち筋

実装（HTML）と GDD v6 を1行ずつ照合し、ユーザー確認済みの確定版。

| キャラ | 勝ち筋（核） | 軸 A/B | winProgress の骨子 |
|---|---|---|---|
| **NEXUS-7** | 量子ビーム起点の属性コンボ（被弾で学習→同属性×2.0） | A 0.65 / B 0.35 | ゲージ＋学習済み属性＋量子バフ窓 |
| **SHADOW** | 速度差→強攻撃回避(ゲージ+25%)＋2回行動確定クリ＋分身斬り | A 0.60 / B 0.40 | 速度優位＋分身斬りゲージ |
| **NOVA QUEEN** | 低HP域で星屑砲バースト逆転（フリップ型） | A 0.70 / B 0.30 | HP低下度×星屑砲射程＋ゲージ |
| **TITAN** | SLOW居座りで硬く殴る→低HPで超電磁砲(保険) | A 0.75 / B 0.25 | SLOWかつ非FAST＋自HP |
| **LYRA-X** | 相手をSLOW→腐食-3＋自分+4/吸収で持久枯らし | A 0.35 / B 0.65 | 相手SLOW（腐食）＋自HP |
| **BLAZE** | 起爆準備で自爆消し→確定クリ爆裂拳56の高火力アグレ | A 0.75 / B 0.25 | 起爆準備ON＋大技を撃てる |

### 三すくみ（設計上の一貫した意図）

- **LYRA ＞ TITAN**：TITAN は自分から SLOW で居座る＝LYRA の腐食の餌になり続ける（TITAN でも -2/turn）。
- **SHADOW ＞ LYRA**：SHADOW は速すぎて SLOW にできない＝腐食が始動しない。テンポでも勝つ。

### 効いてくる重要仕様（照合で確定）

- **強攻撃は速度差20以上で確定スカ**（全キャラ共通）。「戦略的に良さそうでも当たらない手」を採点が救済してはいけない。
- **回復封じ(sealatk)は heal技(+15〜25)・taunt のみを止める。** LYRA の passive **+4HP/turn は封じ無効で常時発動**（`endOfTurn` L6053 は sealTurns を見ない）。`細胞再生` のライフスティール30%が sealTurns で止まるかは実装時に最終確認。
- **速度特攻(SPEED BREAK)** は属性攻撃の技。ボーナスは**撃たれる側の speed が高いほど大**（≥70→+8 / ≥75→+14 / ≥85→+20 / 70未満→0）。dmgMod 後に加算（SHADOW の -15% で削られない）。**自分が速いと強い技ではなく、対FASTの牽制技**。

## 4. コアの仕組み（案1：evalState 拡張）

3つの部品で構成する。

1. **勝ち筋メーター `winProgress(me, opp) → 0〜1`（キャラ別）**
   「このキャラが勝ち筋をどれだけ実行できているか」を返す。**ready で頭打ち**にして溜め込み中毒を防ぐ。

2. **軸の重み表 `AXIS_W[passive] = { a, b }`（キャラ別の小さな係数）**
   `FINISHER_SCALE` と同じ作り。6×6 マッチアップ表ではなく、キャラごとに1組だけ。

3. **採点に2項を追加（evalState 内）**
   既存の nashBuildPayoff が候補手をシミュレーション→結果盤面を evalState で採点しているため、
   この2項を足すだけで二軸の価値が **payoff → regret-matching → 混合戦略** に自動で波及する。
   **新しい判断ルートも、キャラ別の手打ちも不要。**

## 5. 各キャラの winProgress 定義

いずれも 0〜1。各「準備度」項は ready（撃てる状態）で頭打ち＝溜め込みでは軸Aを稼げない。
未設定フラグは 0 として扱い、絶対に例外で停止しない。

```text
NEXUS-7（コンボ準備度）
  0.5×min(gauge/50,1) + 0.3×(learned あり?1:0) + 0.2×(quantum 窓ON?1:0)

SHADOW（速度支配度）
  0.7×clamp((self.speed - opp.speed)/20, 0,1) + 0.3×min(gauge/45,1)
  # 自然回復が無いので、差を保つ手が自然に高評価になる

TITAN（要塞スタンス維持度）
  0.7×(slow && !fast ?1:0) + 0.3×(hp/maxHp)
  # FAST になると第1項が0＝オーバーヒート(弱体化)を自動回避。超電磁砲は既存の固有報酬で保険扱い

LYRA-X（拘束＋持久度）
  0.6×(opp.slow ?1:0) + 0.4×(hp/maxHp)
  # ゲージは勝ち筋でないので入れない（gaugeIncMod 0.8・FINISHER以外で勝つ）

BLAZE（安全火力の構え度）
  0.6×(primed ?1:0) + 0.4×(大技を撃てる?1:0)
  # 生の unprimed 爆裂拳連打は primed項=0＆自HP減で自然に抑制される

NOVA QUEEN（逆転の武装度・フリップ吸収）
  0.4×min(gauge/cost,1) + 0.4×(lostHpFrac)×(星屑砲が撃てる?1:0) + 0.2×(hp ≤ 0.40×maxHp ?1:0)
  # HPが減るほどメーターが上がる＝健康時は仕込み/生存、瀕死で星屑砲レースに跳ねる
```

**NOVA のフリップ簡略化（決定事項）**：当初案は「軸の重み自体を健康/瀕死で切り替える」だったが、
HP依存を winProgress 側に全部入れることで **軸の重みは1組で済む**（同じ挙動・部品が減る）。

## 6. 採点への入れ方（式・重み・スケール）

evalState（自分=me 視点で盤面に点をつける関数）に以下を加算する。

```text
wp_me  = winProgress(me, opp)     // 0〜1
wp_opp = winProgress(opp, me)     // 0〜1
aw     = AXIS_W[me.passive]       // { a, b }

score += AXIS_SCALE × aw.a × wp_me     // 自分の勝ち筋が進むほど加点（軸A）
score -= AXIS_SCALE × aw.b × wp_opp    // 相手の勝ち筋が進んでるほど減点＝止めると相対的に加点（軸B）
```

### 軸の重み表（確定値）

```text
AXIS_W = {
  learning: { a: 0.65, b: 0.35 },   // NEXUS-7
  shadow:   { a: 0.60, b: 0.40 },
  nova:     { a: 0.70, b: 0.30 },
  titan:    { a: 0.75, b: 0.25 },
  lyrax:    { a: 0.35, b: 0.65 },
  blaze:    { a: 0.75, b: 0.25 },
}
```

### AXIS_SCALE（強さの一括ツマミ）

戦略項が盤面価値（HP 等）に対しどれだけ効くかの全体倍率。winProgress は 0〜1 なので、
他の EVAL_W 項（hp:1.0/HP, nearDeath:35, finisher:80 等）と同じ単位。
**初期値は控えめ（目安 25 前後）で「後押し」程度**に置き、6×6 マッチアップ表を見ながら調整する。

### 溜め込み防止

winProgress が「ready で頭打ち」なので、溜めっぱなしで軸A を稼げない。
撃って盤面を有利にしないと点が伸びない。既存の `_finisherBuildup` の保持ガードも維持。

### 二重計上の扱い（段階的）

既存 evalState には `slowTitan` / `fastShadow` / `slowVsLyra` というキャラ別おまけ項があり、
winProgress と意味が一部被る（＝今回の二軸の「原始版」）。

- **初手**：既存おまけ項は**残したまま** AXIS_SCALE を控えめに置いて様子見。
- **被り過剰が出たら**：既存おまけ項を winProgress 側に畳んで撤去（クリーンだが手が増える）。

## 7. ランダム性／難易度

「強いけど毎回同じではない。読み合いで勝てる余地を残す。でも勝てる時は確実に勝つ」を実現する。

- **① 主筋に上限キャップ（核）**：どの1手も確率が **約80%（75〜85%でツマミ化）を超えない**ように頭打ちし、残りを他の手へ配り直す。常に「別の手の枝」を残し、人間の読み・釣りが効くようにする。
- **② リーサル例外**：**今ターンで倒せる（確殺）場面は①のキャップを外して撃ち切る**。変化のために勝ちを取り逃さない。
- **③ 難易度ツマミ（温度）**：確率配分を尖らせる/なだらかにする1つのつまみ。弱い設定はなだらか（人間らしいミス増・倒しやすい）、最上位は鋭く（ただし①のキャップは守る）。

> 補足：現行の EASY/NORMAL は別系統の手打ち CPU（`chooseCpuAction`）。ここの調整が主に効くのは**最上位 Nash を「倒せる手応え」に保つ**部分。

## 8. エッジケース

- **同キャラ対決（ミラー）**：お互い同じ勝ち筋＝メーターも同程度→加点も減点もほぼ釣り合い、素のHP/ダメージ有利で決まる。変な挙動にならない。
- **NOVA が瀕死で突っ込みすぎ自滅**：メーターはHP低下で上がるが、**本当に死ぬ寸前は「生き延びる」評価が最優先**のまま。相手のトドメが見えてるのにガードを捨てて突っ込む、にはさせない（既存のリーサル防御を維持）。
- **LYRA が相手を遅くできない（SHADOW相手など）**：メーターが上がらない→無駄に slow を連発せず、打てる中での最善に切り替わる。SHADOW はそもそも天敵なので、苦しい戦いを最善で戦うで正しい。
- **戦略的に良さそうでも当たらない手**：例＝速度差20以上の相手への強攻撃。一手シミュでスカると盤面が良くならない→戦略の加点があっても低評価。**当たらない手を加点で救済しない。**
- **データ欠損**：学習/腐食/起爆などのフラグ未設定でもメーターは 0 扱い。例外で停止しない。

## 9. 依存・スコープ

- **依存**：`evalState`、`winProgress`（新規）、`AXIS_W`（新規）、`nashBuildPayoff`／`nashSolve`（既存・変更なし）、最終サンプリング段（キャップ/温度を追加）。
- **スコープ外（決定 A）**：既存ヒューリスティック CPU（`chooseCpuAction` 等）に散在する `novaVsLyra` の手打ちは**触らない**。新 Nash 側にも **novaVsLyra のような手打ちを足さない**。
  - 背景：旧ハードモード CPU で NOVA が LYRA に勝てず、手当てが積み上がった経緯がある。二軸モデルに LYRA の winProgress（持久＝+4/turn の止められない固定ドレイン）を入れれば、「LYRA 相手は持久を上回るバーストでレースする／heal技だけ回復封じで切る」が**どのキャラでも原理から創発**し、手打ちは不要になる。

## 10. チューニングつまみ

- `AXIS_SCALE`：戦略項全体の強さ（初期 25 目安）。
- `AXIS_W[passive].{a,b}`：キャラ別の自己促進/相手抑制バランス（§6 確定値が初期）。
- 各 winProgress 内の係数（例 NEXUS 0.5/0.3/0.2）と ready 閾値（gauge/50 等）。
- 主筋キャップ上限（75〜85%）。
- 難易度温度。

## 11. 合格条件

### A. 行動チェック（各キャラが勝ち筋を打てているか）

対戦ログ/プレイで以下が観測できれば合格：

- **NEXUS**：量子ビームで起点を作り、学習属性でコンボを決める（ゲージ溜めっぱなしにしない）。
- **SHADOW**：速度差を保つ／当たらない強攻撃を投げない／相手の SPEED BREAK を読んで COUNTER を置く。
- **NOVA**：低HPに向けて運び星屑砲で逆転を狙う／LYRA 相手に slow・速度特攻を無駄打ちしない（**手打ちなしで自然に**）。
- **TITAN**：遅いまま居座る（加速して FAST=自滅しない）／低HPで超電磁砲。
- **LYRA**：相手を SLOW にして腐食を回し、自分は +4/turn＆吸収で枯らしに行く（殴り合いに付き合わない）。
- **BLAZE**：大技の前に起爆準備を挟む／生の爆裂拳連打で自爆自滅しない。

> マッチアップ事実（LYRA と戦う側向け）：回復封じは heal技しか止めず、LYRA の +4/turn は止まらない。よって相手側は回復封じへ過依存せず、+4/turn を上回るバースト/レースで勝つ。

### B. バランスチェック（数字）

Nash 同士の総当たりを十分な試合数（前回 720 試合の実績あり）回し、
**毎回そろえて出すのは「総勝率」＋「6×6 マッチアップ表」**（固定ルール）。

- 三すくみが表に出る：**LYRA＞TITAN／SHADOW＞LYRA** が各マスではっきり 50% 超。
- 現状の勝率バラつき（最大 62.9%〜最小 37.9%＝約25ポイント差）が**縮む**（目標：15ポイント差以内目安。AXIS_SCALE と重みで詰める）。
- 突出して勝ちすぎ/負けすぎのキャラが出ない。
- **同キャラ対決（対角線）はバラつき判定から除外**（常に約50%）。ただし**サニティチェック**として見る：50% から大きくズレたら「先手番が有利すぎる」等のバグのサイン。

### C. 切り分け・安全

- **二軸 ON/OFF を切り替えた同条件比較**（`AXIS_SCALE=0` と比べる）で今回の変更だけの効果を切り分ける。既存のバッチ集計と NOVA×LYRA 詳細 CSV を流用。
- 勝ちが見える場面でちゃんと勝つ（リーサルを取り逃さない）。
- データ欠損で落ちない。
- 主筋キャップ（約80%）がログ上で効いている（CPU が 100% 予測可能になっていない）。
